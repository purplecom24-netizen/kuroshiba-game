# 要件定義書: 「出来高+陽線」ルール検証システム v1.0

事前登録日: 2026-07-02
以後、判定基準・ルール定義の変更は本書末尾の変更履歴に記録すること。無断変更は検証の無効化とみなす。

## 0. このドキュメントについて

- 本書は Claude Code への実装指示書である。会話の文脈なしで自己完結するよう書かれているため、本書のみを根拠に実装すること。
- 目的は投資判断の提供ではなく、**事前登録された売買ルールの統計的検証**である。仮想資金のみを扱い、証券口座・発注APIへの接続は一切行わない。
- 方式は2段ゲート: **Phase 1 バックテスト(過去5年)** に合格した場合のみ **Phase 2 フォワードテスト(日次スキャン)** を実装する。

## 1. 仮説

- 対立仮説 H1: 「出来高急増を伴う陽線の翌営業日寄付で買う」ルールは、コスト控除後もランダムエントリー(同一出口)および TOPIX を上回る成績を持つ。
- 帰無仮説 H0: 同ルールの成績はランダムエントリーの成績分布の範囲内である(=エッジなし)。

- **頑健性チェック**: 同一テストを中立ユニバース(TOPIX Core30 構成銘柄)でも実行し、両結果を併記する。

### 2.2 シグナル定義(日足・大引け確定後に判定)

- 条件A: 当日出来高 ≥ 直近20営業日の平均出来高(**当日を含まない**) × 2.0
- 条件B: 終値 > 始値(陽線)
- A AND B が成立した日を「シグナル日」とする。
- 除外条件: 上場後の営業日データが20日未満 / 当日または平均算出期間に出来高欠損・0がある / 価格データ欠損。

### 2.3 エントリー

- シグナル翌営業日の寄付(Open)で成行買いを仮定する。**シグナル日当日の終値でのエントリーは先読みバイアスであり禁止。**
- スリッページ: 約定価格 = 寄付 × (1 + SLIPPAGE_PCT)。デフォルト SLIPPAGE_PCT = 0.3%。
- **エントリー無効化条件**:
  - 翌営業日の寄付 ≤ ストップ価格(シグナル日安値)の場合、エントリーしない(シグナル失効)。
  - (エントリー価格 − ストップ価格) / エントリー価格 < MIN_STOP_DIST_PCT(デフォルト0.5%)の場合、ポジションサイズが非現実的に膨らむためスキップ。
- 同一銘柄の重複建て禁止(保有中に同銘柄の新規シグナルが出ても無視)。
- 同日に複数銘柄でシグナル成立時: 最大同時保有 3 ポジション。空き枠を超える場合は「出来高倍率(当日出来高 ÷ 20日平均)」の高い順に採用する。乱数を使わず決定的であること。
- ロングオンリー。空売りなし。レバレッジなし。

### 2.4 出口(最先着優先)

1. **損切り**: ストップ価格 = シグナル日の安値。保有中のある日の安値 < ストップ価格となった場合、ストップ価格で約定と仮定する。ただしその日の**始値がすでにストップ価格を下回っていた場合(ギャップダウン)は始値で約定**(不利な方を採用)。
2. **利確**: 目標価格 = エントリー価格 + 2R(R = エントリー価格 − ストップ価格)。ある日の高値 ≥ 目標価格で目標価格約定。始値が目標を上回っていた場合は始値で約定。
3. **同一日に損切りと利確の両条件に到達した場合: 損切りを優先**(保守的仮定)。この仮定はレポートに明記する。
4. **時間切れ**: エントリー日を1日目として5営業日目の終値で決済。
- エントリー当日も損切り・利確判定の対象とする。

### 2.5 サイズとコスト

- 初期資金: 1,000,000円(仮想)。
- 1トレードのリスク = 決済済みトレードを反映した現在資産の1%。
- 株数 = リスク額 ÷ (エントリー価格 − ストップ価格)。
- ポジション金額はその時点の現金余力を超えないこと。超える場合は余力内に縮小する。
- デフォルトは端株可(理論値モード)。加えて UNIT_CONSTRAINT=True(100株単元制約、建てられない場合スキップ)でも実行し、両モードをレポートする。
- 取引コスト: 片道0.1%(往復0.2%)+ 上記スリッページ。税金は考慮しない(検証目的)。
- 資産は決済ごとに複利更新。

## 3. ベースライン(帰無仮説の実装)

### 3.1 ベースラインA: TOPIX バイ&ホールド

- 同一期間、1306.T(TOPIX連動ETF)を期間初日の寄付で全額買い、期間末の終値で評価する。

### 3.2 ベースラインB: ランダムエントリー × 同一出口(モンテカルロ 1000試行)

- 各試行: ルールの実トレード数と**同数**のトレードを生成する。
- エントリー: ユニバース内の銘柄と営業日を一様ランダムに選び、翌営業日寄付で買う(執行・スリッページはルールと同一)。
- ストップ: エントリー前日の安値(ルールにおけるシグナル日安値に相当する構造)。
- 出口・サイズ・コスト・同時保有制約: ルールと完全に同一。
- 乱数シード: 42 で固定(再現性)。
- 出力: 各試行の最終資産と期待値(R)の分布、その分布に対するルール成績のパーセンタイル位置。

## 4. 判定基準(事前登録・実行前に固定)

### Gate 1(バックテスト合格条件。以下すべてを満たすこと)

1. トレード数 ≥ 100(未満の場合は合否を出さず「検出力不足・判定不能」と結論する)
2. コスト控除後の期待値 > 0(R建て)
3. 最終資産がランダム分布(ベースラインB)の **95パーセンタイル以上**
4. 最大ドローダウン ≤ 25%

- Gate 1 不合格 → ルール棄却。Phase 2 は実装しない。
- パラメータを変更して再テストする場合、それは「別の新ルール」であり、同一データで良い結果が出るまで探索する行為は過剰適合(データスヌーピング)であることをレポートに明記する。

### Gate 2(フォワードテスト)

- 30トレード到達で中間レビュー、50トレードで本レビュー。
- 比較対象: 同期間の TOPIX と、同期間データで再生成したランダム分布。

- Python 3.10+ / ライブラリ: yfinance, pandas, numpy, matplotlib

## 7. データ検証(本体実行前に必須)

- 銘柄ごとに: 期間内の期待営業日数に対する欠損率 / 出来高0の日数 / 日次リターン±30%超の異常値リスト(分割調整ミスの検出)
- auto_adjust による分割・配当調整の妥当性確認(過去に分割のあった銘柄で価格の連続性をチェック)
- 結果を `output/data_quality.md` に出力し、重大な異常があれば本体を実行せず停止する

## 8. 実装の受け入れ基準(QA要件)

1. **先読みバイアス検査**: シグナル判定・エントリー・出口のすべてが当日以前のデータのみに依存すること。検証方法: データ末尾をN日切り落として再実行し、残存期間のシグナル・トレードが完全一致すること
2. **出口ロジック単体テスト**: 合成データで最低3ケース — ①ギャップダウン損切り(始値約定) ②同日両到達(損切り優先) ③時間切れ決済 — を検証
3. **再現性**: 同一入力・同一シードで出力が完全一致すること
4. **サイズ計算テスト**: MIN_STOP_DIST スキップ / 現金余力超過時の縮小 / 単元制約モードの3ケース

## 9. Phase 2: daily_scan.py(Gate 1 合格時のみ実装)

- 平日 18:00 JST に cron 実行(crontab 設定例を提示すること)
- 処理: 当日終値・出来高を取得 → シグナル判定 → `forward_trades.csv` へ記録 → 保有ポジションの出口判定と台帳更新
- 週末に `summary_weekly.md` を自動生成
- 30 / 50 トレード到達時にレビュー用レポートを出力

## 10. スコープ外・禁止事項

- 実際の証券口座・発注APIへの接続
- Gate 判定基準・ルール定義の事後変更
- 同一データ上でのパラメータ最適化の結果を「確認的」成績として報告すること
- LLM による裁量的な銘柄評価・ニュース解釈の混入(本システムは完全ルールベース)

# 自動トレード・ダッシュボード 構築指示書（Claude Code 用）

## 0. このドキュメントについて

- これは Claude Code に渡して**段階的に**実装してもらうための仕様書です。
- フェーズ順に進めること。各フェーズの「完了条件」を満たしてから次へ。
- 「要決定事項」(§4) はユーザー（依頼者）に確認してから着手すること。
- 不明点を勝手に実装で埋めず、ユーザーに質問すること。

-----

## 1. ゴール（何を作るか）

個人用の自動トレード・ダッシュボード。1画面で完結する体験を目指す。

- 銘柄を選ぶ → チャート（ローソク足）が見られる
- 同じ画面に注文パネルがあり、手動でも発注できる
- 設定したルールに従って、システムが自動でシグナル検出・発注を行う
- ユーザーは基本「見ているだけ」。手を動かすのは、画面の監視／ストップの設定・調整／入出金のみ

要するに「監視と最終的なリスク操作は人間、執行は自動」のセミオート・トレーディング環境。

-----

## 2. 重要な前提・制約（最初に必ず読むこと）

この仕様の根幹。実装でここを曲げないこと。

1. **「絶対にマイナスにならない発注」は実装できない。** 相場に損失ゼロを保証するエントリーは存在しない。ストップは滑り（ギャップ・薄商い・フラッシュクラッシュ）、逆指値はトリガーされても約定価格を保証しない。このシステムは**損失の限定**（ストップ／建値ストップ／トレーリングストップ）を行うものであって、**損失の排除ではない**。UI・ドキュメント・ログのどこにも「損しない」「ノーリスク」と表示しないこと。
1. **稼働は必ず バックテスト → ペーパートレード（仮想資金）→ 少額の本番 の順。** ペーパーで戦略が検証されるまで、実資金のAPIキーを接続しない。フェーズを飛ばさない。
1. **リスク制御を必須実装。** 1トレード最大損失、日次最大損失、最大同時ポジション数、キルスイッチ（新規停止＋全決済）。これらが機能しない状態では本番モードを有効化できないようにする。
1. **入出金はアプリで行わない。** 資金移動は証券会社の公式画面でユーザー自身が行う。アプリから出金・送金APIは実装しない（誤発注・不正アクセス時の被害を限定するため）。

-----

## 3. ユーザーがやること／やらないこと

- **やること**：画面の監視／ストップの設定・調整（自動ストップの手動オーバーライド可）／証券会社側での入出金／本番モードの ON-OFF 判断。
- **やらないこと（＝システムが自動でやる）**：シグナル検出／発注／ストップの初期設置と建値移動／ポジション管理／ログ記録。

-----

## 4. 要決定事項（実装前にユーザーに確認）

着手前にこの4点を確定すること。

1. **証券会社／API。** グローバル銘柄重視なら Interactive Brokers（IBKR）。米株中心で手軽さ重視なら Alpaca。※ライブラリ・API 仕様は実装時に最新の公式ドキュメントで確認すること（推奨ライブラリは変わりうる）。
1. **戦略の向き（“売りに入る” の定義）。** 元の要望「一定の値動きがあったものを売りに入る」は2通りに解釈できる：(a) 行き過ぎに対する新規の逆張り（空売り含む）、(b) 既存ポジションの利確（決済売り）。どちらか、または両対応か。戦略エンジンはパラメータで切替可能に作るが、初期デフォルトはユーザーが決める。
1. **初期ユニバース（対象銘柄）。** 「全世界の銘柄をリアルタイム監視」はデータ費用・APIレート・処理負荷が非常に大きい。最初は監視リストを限定（例：主要指数構成銘柄＋ウォッチリスト数十〜数百）で始め、後から拡張できる設計にする。最初の対象範囲をユーザーに確認。
1. **稼働環境。** 自動売買は常時起動が前提。自宅PC常時起動か、クラウドVM（小さな Linux インスタンス等）か。再起動時の自動復帰・状態復元も設計に含める。

## 5. アーキテクチャ

構成要素：

- **Frontend（ダッシュボード）**：チャート、ウォッチリスト、注文パネル、ポジション／損益表示、稼働状況・ログ表示。
- **Backend API**：フロントと各サービスの仲介（REST ＋ WebSocket でリアルタイム配信）。
- **Market Data 層**：価格データの取得・キャッシュ（リアルタイム＆ヒストリカル）。
- **Strategy Engine**：ルール評価とシグナル生成（パラメータ駆動）。
- **Risk Manager**：ストップ計算、建値移動、トレーリング、ポジションサイジング、キルスイッチ、日次損失上限。
- **Broker Adapter**：証券会社APIの抽象化。`paper` と `live` を同一インターフェースで切替。バックテスト用の `sim` 実装も用意。
- **Scheduler / Runner**：自動売買ループ、ジョブ管理、クラッシュ復帰。
- **Store**：SQLite（個人用なら十分）。
- **Logging / Alerts**：全シグナル・発注・約定・エラーを記録。重要イベントは任意で通知（メール／Slack 等）。

推奨スタック（変更可、実装時に最新を確認）：

```
Backend / Bot : Python（pandas, データ処理, 戦略, ブローカーAPI）
Frontend      : React + 軽量チャートライブラリ（例: lightweight-charts）
Realtime      : WebSocket（FastAPI 等）
DB            : SQLite
Backtest      : 自前ループ or 既存バックテストフレームワーク
```

## 6. 段階的ビルド計画

### Phase 0 — セットアップ

- ディレクトリ構成、依存管理、環境変数（APIキーは `.env`、リポジトリにコミットしない）、`paper`/`live`/`sim` のモード切替の枠組み。
- **完了条件**：空のアプリが起動し、モード設定が読み込める。

### Phase 1 — データ＋チャートUI（読み取り専用）

- 銘柄検索、ウォッチリスト、ローソク足チャート（時間足切替）、基本インジケータ表示。
- まだ発注機能なし。価格が見えて更新されることが目標。
- **完了条件**：選んだ銘柄のチャートがリアルタイム（または定期更新）で表示される。

### Phase 2 — バックテスター

- ヒストリカルデータに対して戦略ルールを実行し、結果（損益曲線、勝率、最大ドローダウン、損益比、取引回数）を出す。
- リスク管理ルール（ストップ・建値・トレーリング・サイジング）もバックテストに反映。
- **完了条件**：任意の銘柄・期間・パラメータでバックテストが回り、指標とトレード一覧が出力される。

### Phase 3 — ペーパートレード（仮想資金の自動売買）

- Strategy Engine ＋ Risk Manager ＋ Broker Adapter(`paper`) を接続し、リアルタイムで自動売買。
- ダッシュボードに稼働状況・現在ポジション・損益・ログをライブ表示。注文パネルから手動発注・手動ストップ調整も可能に。
- キルスイッチ・日次損失上限・最大ポジション数を実装し、実際に作動することを確認。
- **完了条件**：仮想資金でルール通りに自動発注・自動ストップ・自動建値移動が動き、リスク制御が機能する。

### Phase 4 — 本番（少額・手動有効化）

- Broker Adapter を `live` に切替（実資金）。ただしユーザーが明示的に ON にしないと本番発注しない。最初は最小ロット。
- 本番モードでも全リスク制御が有効であること。本番 ON の前に、ペーパーの検証基準（§9）を満たしていること。
- **完了条件**：少額・実資金で1サイクル（シグナル→発注→ストップ管理→決済）が安全に完走し、ログに正しく記録される。

## 7. 機能仕様（詳細）

**チャート／ウォッチリスト**：ローソク足、時間足切替、ウォッチリストの追加・削除、銘柄検索、基本インジケータ（移動平均等）。

**注文パネル**：成行／指値／逆指値、数量指定、現在ポジションと評価損益の表示、ストップの手動調整（システムの自動ストップを人間が上書きできる）。

**Strategy Engine（パラメータ駆動）**：

- エントリー条件 “一定の値動き” を具体パラメータで定義（例：直近 N 本での変化率 X% 超、ブレイクアウト、出来高条件 等）。すべて設定可能に。
- 戦略の向き（§4-2）を切替可能。
- 同一シグナルの重複発注防止、クールダウン。

**Risk Manager**：

- 1トレードあたり最大損失（口座資産の％ or 固定額）からポジションサイズを逆算。
- 初期ストップの自動設置。
- 建値ストップ：含み益が一定（例：1R）に達したらストップを建値（または建値＋α）へ移動。
- トレーリングストップ：以降は値動きに追従。
- 日次最大損失に達したら新規停止。最大同時ポジション数の上限。
- キルスイッチ：ワンクリックで新規停止＋全決済。

**スキャナー／ユニバース**：監視対象を設定リストで管理。対象を絞った定期スキャンでシグナル候補を抽出。将来的に対象拡大できる構造（データ費用・レート制限に注意）。

**ログ／監視**：全シグナル・発注・約定・拒否・エラーを時刻付きで記録。ダッシュボードに稼働ステータス（接続状態、最終ハートビート）を表示。任意で重要イベント通知。

-----

## 8. データモデル（最低限・SQLite）

`instruments`（銘柄）/ `candles`（価格キャッシュ）/ `strategies`（戦略とパラメータ）/ `signals`（生成シグナル）/ `orders`（発注）/ `positions`（保有）/ `trades`（約定済み・損益）/ `equity`（資産推移）/ `logs`。

-----

## 9. 検証基準（本番前ゲート）

ペーパートレードで以下を満たすまで `live` を有効化しない（具体数値はユーザーと合意して設定）。

- 一定期間・一定取引回数を完走（少数サンプルで判断しない）。
- バックテストとペーパーの挙動が大きく乖離していない。
- キルスイッチ・日次損失上限・最大ポジション数が実際に作動することを確認済み。
- 約定スリッページやエラー時のハンドリングが想定通り。

-----

## 10. 技術的な現実（ユーザーに伝えるべき注意）

- **データ費用とレート制限**：リアルタイムの全世界データは高コスト＆高負荷。対象は段階的に拡張する。
- **常時稼働の信頼性**：自動売買は止まると機会損失や管理漏れにつながる。再起動復帰・状態復元・接続監視を設計に含める。
- **約定は保証されない**：逆指値・成行はスリッページ前提。
- **口座要件**：API 利用や対象市場により、最低入金額・データ契約が必要な場合がある（実装時に確認）。

-----

## 11. 非目標（やらないこと）

- アプリからの入出金・資金移動。
- 「絶対に損しない」「ノーリスク」を謳う表示やロジック。
- 税務・法務・投資判断のアドバイス機能。
